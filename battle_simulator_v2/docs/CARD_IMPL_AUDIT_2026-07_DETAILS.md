### ENG-1 エールステップ／ライフ処理のエール付けで onCheerAttached 不発 — 種別: 誤実装 / 確度: 高
- ルール: 「このホロメンにエールが付いた時」系の装着トリガーは、エールが付くあらゆる経路で発火すべき
- 実装: `context.js:772 attachCheer()` はフック `attached.onCheerAttached` を発火するが、

## 1. 「このホロメン」対象が「自分のホロメン全員/任意の1人」に化ける

- 種別: 誤実装 / 確度: **高**
- 該当コード: `battle_simulator_v2/core/effects/text-compiler.js:156`（parseSelfTarget）
- 根拠: カードテキストの「このホロメン」は効果の発生源ホロメン自身を指す（4.4 実体、10.3.4.1）。
- 差分説明: `parseSelfTarget` は `spec === 'このホロメン'` を `自分のホロメン`（無条件 `() => true`）と同一扱いにしている。
  `ctx.sourceHolomem` への限定が無いため、下流の全ランナーで対象が誤る:
  - `turnArtsPlus`（text-compiler.js:883）: 「このターンの間、このホロメンのアーツ+20」（hSD17-009 星街すいせい コラボエフェクト）が
    `single=false` 分岐に入り**自分のステージ全員のアーツ+20**になる。
  - `cheerDeckTopTo`（:636）: 「エールデッキの上から1枚を、このホロメンに送る」（hBP04-071 大空スバル アーツ）で
    **任意の自分ホロメンに送れて**しまう。
  - `archiveAttach`（:714）: 「アーカイブのマスコット1枚をこのホロメンに付ける」（hSD14-008 白上フブキ）、
    「アーカイブの〈ルーナイト〉1枚をこのホロメンに付けられる」（hBP05-017 姫森ルーナ）で**任意のホロメンに付けられる**。
- 影響（現行）: hSD17-009 / hBP04-071 / hSD14-008 / hBP05-017 の4種で実害あり（いずれもプレイヤー有利方向のズレ）。
  同テキスト（「このターンの間、このホロメンのアーツ+N」だけで11種・23枠）の他カードは手書き定義が優先されるため潜在。
- 修正方針: 'このホロメン' は `{ make: (ctx) => (e) => e.holomem === ctx.sourceHolomem }` を返す。

## 2. archiveToHand: フィルタが捨てられ、アーカイブの全カードが戻せる

- 種別: 誤実装 / 確度: **高**
- 該当コード: `battle_simulator_v2/core/effects/text-compiler.js:385-388`（パターン）、`:700-713`（ランナー）
- 根拠: テキスト「自分のアーカイブの〈限界飯〉1枚を手札に戻せる」等は対象カード種を限定する。
- 差分説明: パターン側で `parseCardFilter(m[1])` を検証だけして **step に `filter` を入れ忘れている**
  （`return { kind: 'archiveToHand', n, optional }`）。ランナーは `step.filter ? ... : true` で全カード許可にフォールバックするため、
  エール・サポート含む**アーカイブの任意のカード**を手札に戻せる。
- 影響（現行）: hBP04-035 一条莉々華（〈限界飯〉限定のはず）、hBP04-041 カエラ（ツール限定のはず）の2種で実害。
  同パターンのテキストは57種あり、他は手書き定義で回避されているが、手書き削除・新カード追加で即再発する。
- 付随バグ（同パターン内・現状は手書きカードのみのため潜在）: 範囲「1～4枚を手札に戻す」（強制型）で
  `n = m[3]||m[2]`（=最大値）・`optional=false` となり、**1～4枚の任意選択ではなく4枚固定強制**になる。min/max を保持し
  min 枚までは強制・以降任意にすべき（1.3.5）。

## 3. ◆「1st以上の〈X〉に付いていたら能力追加」の Bloom レベル条件を無視

- 種別: 誤実装 / 確度: **高**（コード上明白。現行影響は無し）
- 該当コード: `battle_simulator_v2/core/effects/text-compiler.js:1218-1233`（compileAttached）
- 根拠: テキスト「◆1st以上の〈アキ・ローゼンタール〉に付いていたら能力追加」は付け先が 1st または 2nd であることが条件（2.3 Bloomレベル）。
- 差分説明: 正規表現で `(?:1st以上の)?` をマッチだけして捨てており、条件判定は `h.stack[0].name === name` の**名前一致のみ**。
  Debut の〈X〉に付けても HP+N / アーツ+N が乗る。
- 影響（現行）: 該当12種はすべて手書き定義があり潜在。ただしコンパイラのユニットテスト追加時・新カードで発現する。
- 付随（同関数・低確度）: `attached.hpPlus` は行1187・1225 とも**代入で上書き**（artsPlus は加算チェーンなのに非対称）。
  無条件HP+と◆HP+を両方持つカードが来ると片方が消える。`damageDelta` も同様に上書き。
