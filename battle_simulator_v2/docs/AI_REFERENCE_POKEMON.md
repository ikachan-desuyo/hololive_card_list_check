# ポケカAI（../pokemon-tcg-ai-battle）からの移植分析（2026-07-17）

隣接リポジトリ `pokemon-tcg-ai-battle`（Kaggle Pokémon TCG AI Battle Challenge 向け・Python）の
CPU AI 資産を精読し、ホロライブシミュレーターの AI（core/ai/）へ何が移植できるかを整理した。

## 向こうの構成（要点）

| 層 | 内容 | 規模 |
|---|---|---|
| UniversalBot | **デッキリスト＋カードデータからゲームプランを自動導出**（主役アタッカー・エネ配分規則・土台優先度・開幕戦略）。デッキ固有コードなしで回す | 354行 |
| DeckBot | 設定駆動の Decision Kernel（処理順: 特性→展開→進化→エネ→攻撃、各種ゲート・価値表） | 4,031行 |
| SearchBot | ロールアウト探索。**決定化**（自分の非公開領域はデッキリストから再シャッフル、相手はプレースホルダ）＋**勝率目的**（終局まで再生して勝ち数） | 192行 |
| ValueModelBot | 1手先＋学習価値モデル（状態→勝率のロジスティック回帰）で高速評価 | 84行 |
| 改善ループ | UniversalBot(推論)↔ReplayReviewer(観測)の双対。Episode方式・**修正と採用の分離（採用は実ラダー勝率のみで判定）**・デッキ固有知見の汎用還元（蒸留） | docs/research |

## ホロライブ側に既に同等物があるもの

- ロールアウト探索 = LookaheadAI（再生方式。今日の地平線修正・応手分布平均で強化済み）
- ラダー計測 = tests/abtest.html（凍結ベースライン比勝率）/ selfplay.html（症状メトリクス）
- 決定監査 = tests/ai-audit.html（今日新設。向こうの ReplayReviewer に相当する役割）
- 「採用は勝率で判定」の運用 = 今日のA/Bテスト運用そのもの

## 移植価値が高い順（提案）

### 1. ゲームプラン層（UniversalBot の中核・最有力）
デッキから試合前に「主役アタッカーのライン・必要エール色・土台の優先度」を自動導出し、
序盤の方向付け（どのDebutを置く/センターに据える/どの色をどのラインに貯める/どのBloomを急ぐ）に使う。
- **ホロライブ側には素材が既にある**: `core/ai/firepower.js`（デッキ火力解析）が「対戦AIには組み込まない独立ツール」
  として存在し未統合。これをAI初期化時に走らせて plan 化するのが自然な実装
- 効果が出る場所: 決定監査で最もブレていた**序盤の決定**（エール配分・展開順）に一貫した方向を与える
- 実装案: `core/ai/gameplan.js` 新設 → 自分の全カード（デッキ+手札+ステージ+アーカイブ＝デッキリスト知識で適法）から
  主役ライン上位2つと各アーツのコスト色を導出 → score.js の scoreCheerTargets / bloom / collab / placementCenter に
  「プラン一致ボーナス」を加点（カード番号のハードコードなし・データ由来）。A/Bテストで採否判定

### 2. Decision Gate（用途ゲート）
「Boss(呼び出し)はKOを生む時のみ」「回収は価値がある時のみ」の型。ホロライブでは
『じゃあ敵だね』等の妨害LIMITEDを「引きずり出してKOできる時のみ」使う等、キーカード数枚に効く。
`ai.value`（状況関数）は既にフックがあるので、テキストパターン＋汎用ゲートで実装可能。

### 3. 決定化（determinization）— 将来の大物
全情報依存（相手手札・山札順・未来の乱数を見る）をやめる正攻法。SearchBot は
「自分の非公開領域＝デッキリスト残から再シャッフル、相手＝プレースホルダ」で実現している。
**ホロライブ側の障壁**: エンジンがスナップショット非対応（決定列の再生方式）のため、
「現在状態から未知領域だけ差し替えて分岐」ができない。状態クローン機構の追加が前提＝大工事。
やる価値はある（カオス感度の根治＋対人でも公平なAIになる）が、専用の回で。

### 4. 勝率目的ロールアウト / 学習価値モデル — 実験候補
- 終局まで再生して勝敗で採点（SearchBot型）: 評価関数バイアスに強いが計算コストが跳ねる（現行は3ターン地平線）
- ValueModelBot型（1手先＋学習した状態→勝率モデル）: selfplay で対局データを貯め、
  evaluateState の重み（WEIGHTS）をロジスティック回帰で学習し直す、が最小の第一歩

### 5. 方法論（そのまま採用）
- **Episode方式**: 1改善=1仮説=1計測。修正の実装と採用を分離し、採用は勝率のみで判定
- **蒸留**: デッキ固有の知見（○○デッキで勝てた判断）を、カードデータから導出できる汎用則へ還元して本体に戻す
- 今日の「決定監査→原因特定→修正→A/B」ループは既にこの形。継続の指針として docs に固定する

## 移植できない/しないもの

- DeckBot本体（4,031行）: ポケカ固有の文脈（進化段階・エネタイプ・サポ/グッズ区分・にげる等）が前提。
  コード移植は不可、概念（設定駆動・ゲート・価値表）のみ参考
- ladder_plans / 各デッキ専用bot: デッキ固有コードはホロライブ側の設計原則（カード固有知識は cards/*.js へ）と競合しない範囲で不要
